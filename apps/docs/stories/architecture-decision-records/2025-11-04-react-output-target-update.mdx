# 2025-11-04: React Output Target Update

🗓️ 2025-11-04 · ✍️ [@felipefialho](https://twitter.com/felipefialho_)

## Context

For a long time the project was locked to `@stencil/react-output-target@0.5.3`. That version generated a local `react-component-lib` folder with our own copies of the wrapper helpers (`createComponent`, `createOverlayComponent`, `attachProps`, etc.). Because we could not move forward, we accumulated two custom build-time patch output targets to fix bugs in that generated code:

- `reactBooleanFixOutputTarget` (see ADR 0010): rewrote `attachProps.ts` so boolean props followed the HTML spec (`disabled=""` / attribute removed) instead of being serialized as the strings `"true"` / `"false"`, which Web Components read as truthy.
- `react19FixOutputTarget` (see ADR _Why React 19 compatibility patch?_): rewrote `createOverlayComponent.tsx` and `utils/index.tsx` so the generated `forwardRef` usage compiled under React 19.

Both patches were workarounds for limitations of the old generator, and both depended on a locally generated `react-component-lib` that only the `0.x` line produces. This made every dependency bump risky and blocked React 19 adoption ([output-targets#552](https://github.com/stenciljs/output-targets/issues/552), [output-targets#645](https://github.com/stenciljs/output-targets/issues/645)).

## Decision

Update to `@stencil/react-output-target@^1.5.3` and adopt its runtime-based API, removing both custom patch output targets.

### 1. Stencil configuration

```typescript
reactOutputTarget({
  outDir: '../react/src/components',
  stencilPackageName: '@juntossomosmais/atomium',
  customElementsDir: 'dist/components',
  excludeComponents, // Ionic components remain excluded
  esModules: true,
})
```

The Ionic components stay in the `excludeComponents` list, exactly as before, so we do not emit React wrappers for raw `ion-*` elements. Atomium's own components (which wrap Ionic internally) are generated against the `dist-custom-elements` output. `esModules: true` makes the generator emit one file per component plus a `components.ts` barrel, which is what enables per-component tree-shaking (see section 5).

### 2. Runtime instead of a local `react-component-lib`

The `1.x` generator no longer emits a local `react-component-lib`. Each component is generated as its own file (`atom-button.ts`, `atom-input.ts`, ...) plus a `components.ts` barrel, and every file imports from the published runtime:

```typescript
import { createComponent } from '@stencil/react-output-target/runtime'
import type { EventName, StencilReactComponent } from '@stencil/react-output-target/runtime'
```

Internally that runtime delegates to [`@lit/react`](https://www.npmjs.com/package/@lit/react), which assigns values as DOM **properties** (`el.disabled = false`) rather than stringified attributes.

### 3. Both custom patches are now obsolete

- **Boolean fix is no longer needed.** Because `@lit/react` sets properties instead of attributes, the boolean-as-string bug from ADR 0010 cannot happen.
- **React 19 fix is no longer needed.** There is no local `react-component-lib` to patch, and the runtime declares `react: ^18 || ^19`.

The `packages/core/output-target/` folder (`react-19-fix.ts`, `react-boolean.ts`) was removed. This ADR supersedes ADR 0010 and the _Why React 19 compatibility patch?_ ADR.

### 4. Ionic styling fix (output-targets#552)

The `1.x` model registers components through the `dist-custom-elements` output, with each wrapper importing its own `dist/components/atom-*.js` and calling `defineCustomElement` individually. **This path never runs Stencil global scripts**, so Ionic's own global initialization (which sets the default `mode`) does not fire. Without a resolved `mode`, Ionic's mode-specific stylesheet (`button.md` / `button.ios`) is never attached and Ionic-backed components render unstyled. This is the root cause of the long-standing [output-targets#552](https://github.com/stenciljs/output-targets/issues/552).

The fix is to set the mode resolver explicitly in the core global script, so it is compiled into the shared Stencil runtime chunk every component imports:

```typescript
// packages/core/src/global/global.ts
import { setMode } from '@stencil/core'

setMode(() => 'md')
```

Mode is forced to `md` for consistency across platforms. Atomium is a Material-based design system, so we deliberately do not want per-OS Ionic styling (`ios` on Apple devices). This also makes the Vue / `defineCustomElements()` loader path deterministic.

### 5. Build output, package exports and tree-shaking

`packages/react` compiles with `tsc` into `packages/core/react/dist` and is published through the core package's `./react` export.

- **Flat ESM output.** `packages/react/tsconfig.json` sets an explicit `rootDir: "src"` and `moduleResolution: "Bundler"` (required to resolve the `@stencil/react-output-target/runtime` subpath). It does **not** use `paths` mappings: an earlier attempt mapped the bare `@juntossomosmais/atomium` import to `../core/src`, which dragged core source into `tsc`'s emit set and nested the output (`react/dist/react/src/index.js`), breaking the `@juntossomosmais/atomium/react` resolution. The bare import now resolves through the workspace symlink to core's built declarations instead.
- **ESM-only export.** The `./react` export ships `import` + `types` only (no `require`/CJS condition):

  ```jsonc
  "./react": {
    "types": "./react/dist/index.d.ts",
    "import": "./react/dist/index.js"
  }
  ```

- **`./dist/components` export.** Added to `packages/core/package.json` because the generated proxies import the `dist/components` directory, which the `./dist/*` pattern does not resolve under Node `exports`.
- **Tree-shaking.** The react package declares `"sideEffects": false`. Combined with the per-component files from `esModules: true`, importing a single component no longer pulls the whole library (a single `AtomButton` import dropped from ~880 KB to ~93 KB in an esbuild check).

### 6. Dependency management

`@stencil/react-output-target` is a dev dependency at the root (used to generate the wrappers at build time) **and a runtime `dependency` of the published `@juntossomosmais/atomium` (`packages/core`)** so its `/runtime` import resolves in consumer apps. `react` and `react-dom` are declared as optional peer dependencies. Consuming apps do not need to install the runtime directly.

## Consequences

### Positive

- ✅ No more custom patch output targets to maintain.
- ✅ React 19 supported natively, dependency bumps are no longer blocked.
- ✅ Ionic-backed components render correctly in React again (closes the long-standing #552).
- ✅ Per-component files plus `sideEffects: false` make imports genuinely tree-shakeable.

### Neutral

- 🔄 Internal change in how wrappers are generated (not visible at the import-path level).
- 🔄 `@stencil/react-output-target/runtime` (and `@lit/react`) ship in the bundle.
- 🔄 Ionic `mode` is now always `md`, including on the Vue / loader path (previously platform-adaptive).

### Breaking (major release)

- 💥 **ESM-only `./react` export.** The `require`/CJS condition was removed. Consumers that load the wrappers as CommonJS (notably Jest with `jest-environment-jsdom`, which resolves the `require` condition) must allowlist `@juntossomosmais/atomium` in `transformIgnorePatterns`, or set `testEnvironmentOptions.customExportConditions: ['import']`.
- 💥 **React `>=18` required.** `@stencil/react-output-target@1.x` dropped React 16/17 support.
